# Pure Water OS Hub (Notion Replacement) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. Phases 1–5 are written at task granularity; write the detailed per-task TDD breakdown at the start of each phase (the design files are the pixel-level spec).

**Goal:** Wire the "Pure Water OS Hub" Claude Design (projects-as-pages, block editor, custom fields, scratchpad + Purii extraction, Library, client-portal projection) into `va-management-next`, running on an isolated test environment at `https://dev-projects.pwasecondbrain.uk` until approved for merge into dev → prod.

**Architecture:** The Hub is ~70% a new presentation layer over models that already exist (Project, Task, MeetingAction confirm-pipeline, ClientOrganization/ClientTaskRequest, comments with visibility). The genuinely new data layer is four additive models: `Page` (blocks as JSON), `FieldDef`/`FieldValue` (custom fields), `ScratchItem`, and later `Link`. All migrations are additive-only so the feature branch can merge into `main`/`production` without touching existing tables' shape.

**Tech Stack:** Next.js 15 App Router, React 19, Prisma 5 + Postgres, NextAuth (Google), CSS-variable token system (no Tailwind). AI extraction reuses the existing OpenRouter plumbing from the transcript worker.

**Design sources** (unzipped from `Replacing Notion for task management.zip`):

- `PWA OS Master (All Sprints).dc.html` — Hub + Sprint 2/3 screens (Sprint 2/3 files are identical with toggles pinned)

## Phase 1 — Fields & Views (the design's own "ship first" decision)

Recorded meeting decision in the design: *"Fields & views ship before the doc editor."* Smallest lift, immediately useful, zero editor risk.

**New models** (additive migration via `prisma migrate diff --script` → `migrations/<ts>/migration.sql` → `migrate deploy`; never `migrate dev`):

```prisma
enum FieldType { TEXT SELECT DATE PERSON }

model FieldDef {
  id            String       @id @default(cuid())
  projectId     String?      // null = global (all projects)
  project       Project?     @relation(fields: [projectId], references: [id], onDelete: Cascade)
  name          String
  type          FieldType
  options       Json?        // SELECT choices: ["Research","Build","Launch"]
  clientVisible Boolean      @default(false)
  order         Int          @default(0)
  createdById   String
  createdAt     DateTime     @default(now())
  values        FieldValue[]
  @@unique([projectId, name])
}

model FieldValue {
  id        String   @id @default(cuid())
  fieldId   String
  field     FieldDef @relation(fields: [fieldId], references: [id], onDelete: Cascade)
  taskId    String?  // per-task column value…
  projectId String?  // …or project-level pill value (exactly one is set)
  value     String
  updatedAt DateTime @updatedAt
  @@unique([fieldId, taskId])
  @@unique([fieldId, projectId])
}
```

## Phase 2 — Hub shell + doc pages (block editor)

**New model** — blocks as a JSON column (the concept file's own suggestion; per-block rows are YAGNI at this team size). `version` gives optimistic-concurrency protection for two editors on one page.

```prisma
enum PageScope { PROJECT LIBRARY }

model Page {
  id            String    @id @default(cuid())
  scope         PageScope
  projectId     String?
  project       Project?  @relation(fields: [projectId], references: [id], onDelete: Cascade)
  parentId      String?
  title         String
  order         Int       @default(0)
  blocks        Json      @default("[]")  // [{id, kind: callout|h2|p|todo|ul|ol|code|chip, text, done?, ref?: {type:"task"|"sop"|"video", id}}]
  version       Int       @default(1)     // optimistic lock: save sends version, 409 on mismatch
  published     Boolean   @default(false) // LIBRARY: visible in client portal "Shared with you"
  clientVisible Boolean   @default(false) // PROJECT: rendered read-only as the portal Overview
  createdById   String
  updatedById   String?
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
  @@index([projectId, parentId])
}
```

### Tasks

- [x] **2.1** Migration + `src/lib/services/blocks.ts` — pure block ops (insert/edit/toggle-todo/reorder, slash-command parse, ol renumbering) with unit tests. This is where the TDD lives; the React layer stays thin.
- [x] **2.2** `src/lib/actions/pages.ts` + `src/lib/reads/pages.ts` + `api/hr/pages` (create page/sub-page, save blocks with version check, tree query).

## Testing & promotion path

1. Every phase deploys to **hub** (`./deploy.sh hub`) as it lands — Justin + team test on `dev-projects.pwasecondbrain.uk` against the cloned DB. Use the `run-va-management-next` skill locally before each deploy; pure logic (blocks, fields, extraction, request-status derivation) carries unit tests in `tests/`.
2. **Migration discipline:** additive-only (new tables/columns, no drops/renames/enum-value removals). The same migration files later run against the real dev + prod DBs on merge — this is what makes the sandbox safe to promote.
3. When satisfied: merge `feature/os-hub` → `main` → `./deploy.sh dev` (dev-team). Data note: hub-created content (pages/fields/scratch) lives only in `va_console_hub` — real content gets created fresh on dev/prod (or re-imported from Notion, see enhancement #1). The clone is a throwaway.
4. Production: merge to `production` → `./deploy.sh prod` per house standard. Then decommission the hub instance (stop service, drop `va_console_hub`, remove tunnel ingress) or keep it as the standing "next big thing" sandbox.

## Risks & guardrails

- **Feature parity (standing rule):** the Hub replaces the project detail — checklist, dependencies, task comments, templates, Notion-link chips must remain reachable (hence `classic` route + task-detail links) before the old page is deleted.
- **Cloned DB ≠ inert by default:** sanitization (0.3) is what prevents the test box emailing/WhatsApping real people or double-writing the mirror sheet. Treat it as a hard gate before first login.
- **No second worker fleet:** timers stay uninstalled on the test instance; the transcript pipeline keeps running only against real dev/prod.
- **Two editors, one page:** JSON-blocks + `version` optimistic lock is deliberate minimalism — a 409 shows "page changed, reload". Real-time co-editing is explicitly out of scope.
- **R2 recorder CORS:** recordings upload from the new origin will fail unless the `va-recordings` bucket CORS adds `dev-projects.…` — skip unless recorder testing is needed there.

## Enhancement ideas (beyond the design — for Justin to pick from)

1. **Notion importer (highest leverage):** one-shot import of existing Notion project docs/SOPs into `Page` blocks using the already-built `src/lib/notion.ts` client — the design's Phase "migrate content" currently assumes manual re-typing; an importer turns the Aug-15 cutover from a chore into a button.
2. **MCP tools for the Hub:** add `create_page` / `append_blocks` / `add_scratch_item` to `src/lib/mcp/tools.ts` — then Claude sessions (and the team-Claude box) can write project docs directly, which fits the whole SecondBrain workflow.
3. **Live task chips:** `/task` chips render current status/assignee (they reference task ids, so this is nearly free) — a doc that never lies about task state is the anti-Notion selling point.
4. **Page templates:** `ProjectTemplate` already exists; add a page-template on project type (Event projects get Plan/Run-sheet/Retro pages automatically).
5. **Full-text search on blocks:** Postgres `pg_trgm`/tsvector over `Page.blocks` text so ⌘K finds content, not just titles.
6. **Activity-aware portal:** surface "team updated the Overview 2h ago" in the client portal — cheap trust-builder from existing `ActivityLog`.
7. **Scratchpad quick-capture endpoint:** a tokened `POST /api/hr/scratch/capture` so Telegram/voice-note pipelines can drop thoughts into a project scratchpad from outside the app.
