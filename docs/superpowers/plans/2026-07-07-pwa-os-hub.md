**Goal:** Wire the "Pure Water OS Hub" Claude Design (projects-as-pages, block editor, custom fields, scratchpad + Purii extraction, Library, client-portal projection) into `va-management-next`, running on an isolated test environment at `https://dev-projects.pwasecondbrain.uk` until approved for merge into dev ‚Üí prod.

**Architecture:** The Hub is ~70% a new presentation layer over models that already exist (Project, Task, MeetingAction confirm-pipeline, ClientOrganization/ClientTaskRequest, comments with visibility). All migrations are additive-only so the feature branch can merge into `main`/`production` without touching existing tables' shape.

**Tech Stack:** Next.js 15 App Router, React 19, Prisma 5 + Postgres, NextAuth (Google), CSS-variable token system (no Tailwind). AI extraction reuses the existing OpenRouter plumbing from the transcript worker.

## Phase 1 ‚ÄĒ Fields & Views (the design's own "ship first" decision)

Recorded meeting decision in the design: *"Fields & views ship before the doc editor."* Smallest lift, immediately useful, zero editor risk.

**New models** (additive migration via `prisma migrate diff --script` ‚Üí `migrations/<ts>/migration.sql` ‚Üí `migrate deploy`; never `migrate dev`):

```prisma
enum FieldType { TEXT SELECT DATE PERSON }

model FieldDef {
  id            String       @id @default(cuid())
  projectId     String?      // null = global (all projects)
  project       Project?     @relation(fields: [projectId], references: [id], onDelete: Cascade)
  name          String
  type          FieldType
  options       Json?        // SELECT choices: ["Research","Build","Launch"]
  clientVisible Boolean      @default(false)
  order         Int          @default(0)
  createdById   String
  createdAt     DateTime     @default(now())
  values        FieldValue[]
  @@unique([projectId, name])
}

model FieldValue {
  id        String   @id @default(cuid())
  fieldId   String
  field     FieldDef @relation(fields: [fieldId], references: [id], onDelete: Cascade)
  taskId    String?  // per-task column value‚Ä¶
  projectId String?  // ‚Ä¶or project-level pill value (exactly one is set)
  value     String
  updatedAt DateTime @updatedAt
  @@unique([fieldId, taskId])
  @@unique([fieldId, projectId])
}
```

## Phase 2 ‚ÄĒ Hub shell + doc pages (block editor)

**New model** ‚ÄĒ blocks as a JSON column (the concept file's own suggestion; per-block rows are YAGNI at this team size). `version` gives optimistic-concurrency protection for two editors on one page.

```prisma
enum PageScope { PROJECT LIBRARY }

model Page {
  id            String    @id @default(cuid())
  scope         PageScope
  projectId     String?
  project       Project?  @relation(fields: [projectId], references: [id], onDelete: Cascade)
  parentId      String?
  title         String
  order         Int       @default(0)
  blocks        Json      @default("[]")  // [{id, kind: callout|h2|p|todo|ul|ol|code|chip, text, done?, ref?: {type:"task"|"sop"|"video", id}}]
  version       Int       @default(1)     // optimistic lock: save sends version, 409 on mismatch
  published     Boolean   @default(false) // LIBRARY: visible in client portal "Shared with you"
  clientVisible Boolean   @default(false) // PROJECT: rendered read-only as the portal Overview
  createdById   String
  updatedById   String?
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
  @@index([projectId, parentId])
}
```

## Testing & promotion path

1. Every phase deploys to **hub** (`./deploy.sh hub`) as it lands ‚ÄĒ Justin + team test on `dev-projects.pwasecondbrain.uk` against the cloned DB. Use the `run-va-management-next` skill locally before each deploy; pure logic (blocks, fields, extraction, request-status derivation) carries unit tests in `tests/`.
2. **Migration discipline:** additive-only (new tables/columns, no drops/renames/enum-value removals). The same migration files later run against the real dev + prod DBs on merge ‚ÄĒ this is what makes the sandbox safe to promote.
3. When satisfied: merge `feature/os-hub` ‚Üí `main` ‚Üí `./deploy.sh dev` (dev-team). Data note: hub-created content (pages/fields/scratch) lives only in `va_console_hub` ‚ÄĒ real content gets created fresh on dev/prod (or re-imported from Notion, see enhancement #1). The clone is a throwaway.
4. Production: merge to `production` ‚Üí `./deploy.sh prod` per house standard. Then decommission the hub instance (stop service, drop `va_console_hub`, remove tunnel ingress) or keep it as the standing "next big thing" sandbox.

## Risks & guardrails

- **Feature parity (standing rule):** the Hub replaces the project detail ‚ÄĒ checklist, dependencies, task comments, templates, Notion-link chips must remain reachable (hence `classic` route + task-detail links) before the old page is deleted.
- **Cloned DB ‚Č† inert by default:** sanitization (0.3) is what prevents the test box emailing/WhatsApping real people or double-writing the mirror sheet. Treat it as a hard gate before first login.
- **No second worker fleet:** timers stay uninstalled on the test instance; the transcript pipeline keeps running only against real dev/prod.
- **Two editors, one page:** JSON-blocks + `version` optimistic lock is deliberate minimalism ‚ÄĒ a 409 shows "page changed, reload". Real-time co-editing is explicitly out of scope.
- **R2 recorder CORS:** recordings upload from the new origin will fail unless the `va-recordings` bucket CORS adds `dev-projects.‚Ä¶` ‚ÄĒ skip unless recorder testing is needed there.
