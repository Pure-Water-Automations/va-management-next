Your in-flight decisions

Three design calls you've been weighing. Each is laid out as options with a clearly-marked recommendation (the green card) — pick one to move it forward.

A

Configurable delegation authority by comp tier

Let admin/HR decide, on the Compensation Roles screen, which comp tiers may delegate tasks and/or projects. Default: Tier 3 (Senior VA) + Tier 4 (Lead) can delegate; lower tiers can't. A logged-in VA's authority then resolves through their comp tier.

Compensation roles · delegation authority
Comp tierCan delegate tasksCan delegate projects
Trainee
Tier 1
Tier 2
Tier 3 · Senior VA
Tier 4 · Lead
Recommended

Store flags on CompensationRole

  • Add canDelegateTasks / canDelegateProjects booleans to each comp role
  • Edited right on the comp-roles table (toggles above)
  • Authority resolves: VA → comp role → flags
Why: the data lives where you already manage tiers; one source of truth, easy UI, scales as tiers change.
Option

Global Setting list

  • One Setting like delegating_tiers = [3,4]
  • Fastest to ship; no schema change
  • Less granular (can't split task vs project)
Good stop-gap, but you lose the task/project split and the in-context toggles.
Option

Keep access-role only

  • No change — delegation stays tied to HR/PEOPLE_OPS/TEAM_LEAD/SENIOR_VA
  • Zero effort
  • Comp tier & authority stay decoupled
Status quo. Only if tier-based delegation isn't actually wanted.

Recommendation: flags on CompensationRole, defaulting Tier 3 + Tier 4 on for tasks (Tier 4 only for projects). It keeps tier and authority in one place and gives HR a clean per-tier toggle.

B

All VAs can receive tasks regardless of tier

The assignee picker shows every active VA, not just the handful with manager logins. Prerequisite: VAs need login accounts — today only ~6 manager accounts exist, so most VAs can be assigned to but can't yet log in to see their queue.

Assign to
JO Justinhas login
RZ Rizano login yet
KN Kannano login yet
Recommended

Show all, account on first assign

  • Picker lists every active VA immediately
  • First assignment auto-provisions a login (CF Access + User row)
  • Assignee still gets the email notification meanwhile
Why: unblocks delegation now and grows logins exactly as needed — no big-bang account rollout.
Option

Provision all logins first

  • Create accounts for every VA up front
  • Cleanest end state
  • Heaviest lift; many accounts may sit unused
Right eventually, but slow to start and front-loads work.
Option

Assignable-only (no login)

  • Show all VAs; assignment = email only, no console for them
  • Trivial to ship
  • VAs can't see/update their own queue in-app
Fine as a phase 1 if logins lag, but the queue value is lost until they sign in.

Recommendation: show all active VAs in the picker now, and auto-provision a login on first assignment. Pairs naturally with decision C (VAs need a login to self-add and get notified).

C

Tier 1+ VAs can self-add tasks to a project

Let Tier 1 and above add their own tasks to a project they're part of, with the VA's supervisor notified in-console (a bell/inbox, not email). Open questions: how "supervisor" is determined, and how in-console notifications work.

Recommended

Notification table + bell

  • New Notification rows; a sidebar bell with unread count
  • Supervisor = the project owner (or VA's assignedBy / team lead)
  • Self-add gated to Tier 1+ via comp tier (reuses decision A's resolver)
Why: in-console, no email noise, and reuses the tier-authority plumbing you're already adding.
Option

Email the supervisor

  • Reuse the existing assignment email path
  • Fast — no notification system needed
  • Not what was asked (you wanted in-console)
Quick, but adds inbox noise and misses the bell experience.
Option

Just an activity-feed entry

  • Log the self-add to the project Activity feed only
  • Almost free — feed already exists
  • Passive; supervisor must go look
Lowest effort, but no real "you have something to review" signal.

Recommendation: a lightweight Notification table + sidebar bell, with "supervisor" = the project owner (fall back to the VA's assignedBy/team lead). Reuses decision A's tier resolver to gate who can self-add.
